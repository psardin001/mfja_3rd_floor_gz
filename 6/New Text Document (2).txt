Kinematic Work (Main Progress)

I built splines from the Excel data and extracted two initial clean path sets:

Small loop points
Large loop points

These points were exported to CSV and used as the basis for the shuttle motion path.

In kinematic mode, the controller reads the CSV data and directly sets the shuttle pose along the path (set_pose).

The kinematic controller enforces:

roll = 0
pitch = 0
only yaw changes

Because the CSV contains only X, Y, Z coordinates, yaw is not directly provided.
Instead, yaw is computed from the local path direction:

the heading is estimated from the tangent between consecutive points
Important Observations

Using only X, Y, Z is not sufficient for fully realistic motion.

The shuttle orientation is approximated from the tangent of the path, which leads to:

simplified rotational behavior
reduced physical realism compared to the real shuttle motion on the rail

However, this approximation is currently acceptable for functional validation and early control development.

Technical Implementation Details
CSV points are interpreted in the local frame of room315_cell_path_right_1
A scale and offset are applied around the path center to match the real rail geometry
Update period is clamped to 0.02 s for smooth motion
Dynamic Simulation Attempts

One key idea I attempted was:

applying a force to the shuttle based on its position along the path
using the Excel points as reference targets

The goal was to:

approximate rail motion using force-based control instead of directly setting pose
guide the shuttle progressively from one target point to the next

However, this approach did not converge to a stable solution:

the applied forces produced oscillatory and unstable behavior
the shuttle could not reliably follow the intended path

Based on these results, I decided to pause the dynamic approach for now and focus on a fully kinematic solution.

The objective is to:

first obtain a stable and functional system (paths, switches, control logic, multi-shuttle support)
avoid spending excessive time on unstable physics behavior at this stage

Once the kinematic system is complete:

I will return to the dynamic approach in a separate dedicated branch
and work on improving it with better constraints and structure
Path Modeling Strategy Agreed During Meeting 6

A key point discussed during Meeting 6 was that the system should not be represented as only two fixed loops.
Instead, the motion logic should be built from a set of spline-based admissible paths, where each path corresponds to a specific switch configuration.

The idea is the following:

the rail network topology changes according to the switch states
for each valid switch configuration, a corresponding spline path can be generated or selected
the shuttle is then allowed to move only along the path that is physically admissible for the current switch state

This gives a more consistent kinematic abstraction of the real system:

the shuttle does not move freely in space
it evolves on a constrained geometric path derived from the rail configuration at that moment

From a control perspective, this means:

the switch states define the currently valid route
the shuttle state can be represented by its progress along the active spline
when switches change, the future admissible path changes accordingly

An important consequence of this modeling choice is related to failure behavior.

If, due to an incorrect control program, the shuttle is commanded into a configuration where the selected route is no longer physically connected to a valid downstream rail segment, then the simulation should not artificially keep it on a safe trajectory.

Instead, the model should reflect the real system behavior:

the shuttle can only remain constrained while a valid rail-supported path exists
if it reaches an open or non-supported route, the kinematic constraint should end
from that point, the shuttle should lose support and fall under gravity, as it would in the real installation

This is important because it preserves the pedagogical value of the simulation:

correct switch logic keeps the shuttle on a valid path
incorrect switch logic produces a visible physical consequence

In other words, the simulation should not hide programming errors by automatically correcting the route.
