شرح المنطق العام لمنظومة Room 315

هذا الملف يعطيك خريطة عامة لكيف تتحرك الرسائل داخل الكود: من ينشر، من يشترك، ماذا يحدث عند وصول command، وكيف ترتبط السكة والشاتلات وGazebo والروبوتات.

الهدف ليس شرح كل سطر، بل أن تقرأه وتعرف أين تبدأ التتبع عندما تريد فهم سبب حركة شاتل، تغير حساس، تغير سويتش، أو وصول أمر إلى روبوت.

الخريطة الذهنية العقد التي تعمل التوبيكات والخدمات ماذا يحدث عند التشغيل منطق السويتشات منطق الـ stoppers منطق الشاتلات اللوب الداخلي منطق الحساسات التعامل مع Gazebo منطق الروبوتات كيف تتتبع مشكلة

1. الخريطة الذهنية السريعة

عندك ثلاث طبقات أساسية:

طبقة منطق السكة

موجودة في room_315_kinematic_shuttle_node.py و room_315_kinematic_shuttle.py. هذه الطبقة تحسب أين الشاتل على السكة، أي segment يستخدم، وماذا يحصل عند السويتش أو stopper.

طبقة Gazebo والمرئيات

الشاتل منطقي في الكود، لكن لكي تراه في Gazebo، الكود يرسل pose إلى Gazebo. السويتشات البصرية تتحرك عبر conveyor_loop_mode_controller.py.

طبقة الروبوتات

الروبوتات مثل kuka1 و staubli1 تُضاف من multi_robot_sim.launch.py. التحكم بها يتم عبر توبيكات joint trajectory والجسر بين ROS وGazebo.

User / Controller ينشر أوامر switches, stoppers, shuttles, robot trajectories
Room315 rail node يحسب الحركة ويحدث الحالة وينشر feedback
Gazebo bridge/services ينقل poses ويخلق أو يحذف models ويغير السويتشات المرئية
Consumers واجهة، PLC خارجي، سكربت اختبار، أو روبوت يقرأ feedback ويتصرف

2. العقد التي تعمل

العقدة / الملف متى تعمل دورها
room_315_kinematic_shuttle_node.py room_315_dual_kinematic_shuttles.launch.py تعمل مرتين غالباً: مرة namespace /room_315/rails/right ومرة namespace /room_315/rails/left. هي القلب الذي يحرك الشاتلات وينشر state والحساسات.
room_315_kinematic_shuttle.py يستخدمه الـ node كـ library يحتوي منطق السكة الخام: قراءة rail_network_*.yaml، حساب طول segments، وتحريك الشاتل داخل segment ثم اختيار successor.
conveyor_loop_mode_controller.py من launch العالم الكامل يشترك بأوامر السويتشات البصرية على /mfja/conveyor/switch_cmd، يغير وضع models في Gazebo، ثم ينشر ملخص الحالة على /mfja/conveyor/switch_states.
multi_robot_sim.launch.py عند تشغيل العالم والروبوتات يضيف الروبوتات إلى Gazebo، يشغل robot_state_publisher، وينشئ bridges مثل /kuka1/joint_trajectory و /kuka1/joint_states.

3. التوبيكات والخدمات الأساسية

كل rail لها نفس البنية. استبدل {side} بـ right أو left.

الاسم نوع الرسالة الاتجاه المعنى
/room_315/rails/{side}/switches/command mfja_rail_interfaces/msg/SwitchCommand داخل إلى rail node أمر تغيير سويتش: A1=EXTERIOR أو A1=INTERIOR.
/room_315/rails/{side}/switches/state mfja_rail_interfaces/msg/SwitchState خارج من rail node الحالة الفعلية الحالية بعد delay الحركة، وليست مجرد request.
/room_315/rails/{side}/stoppers/command mfja_rail_interfaces/msg/StopperCommand داخل إلى rail node 1 يعني stop/closed. 0 يعني pass/open.
/room_315/rails/{side}/stoppers/state mfja_rail_interfaces/msg/StopperState خارج من rail node الحالة الفعلية للـ stoppers بعد delay الحركة.
/room_315/rails/{side}/shuttles/command mfja_rail_interfaces/msg/ShuttleCommand داخل إلى rail node تشغيل، إيقاف، reset، أو remove لشاتل موجود.
/room_315/rails/{side}/shuttles/state mfja_rail_interfaces/msg/ShuttleState خارج من rail node حالة الشاتل الأول: segment, s, x/y/z, yaw, mode, speed.
/room_315/rails/{side}/sensors/feedback mfja_rail_interfaces/msg/SensorFeedback خارج من rail node قراءات الحساسات الثنائية: active أو clear مع اسم الشاتل إن وجد.
/room_315/rails/{side}/shuttles/add mfja_rail_interfaces/srv/AddShuttle service داخل rail node إضافة شاتل جديد على slot محدد مع speed وحالة تشغيل ابتدائية.
/mfja/conveyor/switch_cmd std_msgs/msg/String من rail node إلى visual controller أمر بصري مثل A1R=EXTERIOR أو A1L=INTERIOR.
/mfja/conveyor/switch_states std_msgs/msg/String من visual controller إلى rail nodes ملخص الحالة البصرية. الـ rail node يستطيع sync منه عند التشغيل.

الرسالة NamedState بسيطة جداً: فيها name و state. لذلك أوامر السويتش والـ stopper هي arrays من أزواج اسم/حالة.

4. ماذا يحدث عند التشغيل؟

  1. room_315_dual_kinematic_shuttles.launch.py يشغل node للسكة اليمنى وnode للسكة اليسرى حسب enable_right و enable_left.
  2. كل node يحدد rail_side، ثم يختار ملفات YAML المناسبة: rail_network_right.yaml أو rail_network_left.yaml، ومعها rail_devices_right.yaml أو rail_devices_left.yaml.
  3. RailNetwork.from_yaml() يقرأ segments وrouting والسويتشات.
  4. load_rail_devices() يقرأ slots والحساسات والـ stoppers.
  5. إذا shuttle_count أكبر من صفر، ينشئ الشاتلات الابتدائية. في launch الحالي القيمة الافتراضية صفر، لذلك غالباً تضيف الشاتل عن طريق service.
  6. يبدأ timer الحركة حسب update_rate_hz، ويبدأ timer الحساسات حسب sensor_publish_rate_hz.

5. منطق السويتشات

عندما تنشر command

ros2 topic pub --once /room_315/rails/right/switches/command \
  mfja_rail_interfaces/msg/SwitchCommand \
  "{switches: [{name: 'A1', state: 'EXTERIOR'}]}"
  1. الرسالة تصل إلى _on_switch_command().
  2. الكود يحول الاسم والحالة إلى منطق داخلي. القيم المقبولة: E EXTERIOR I INTERIOR
  3. _schedule_switch_state_updates() لا يغير الحالة فوراً إذا switch_motion_delay_s أكبر من صفر. يضع update pending.
  4. عند انتهاء delay، _apply_due_pending_state_updates() يضع الحالة في self.switch_states.
  5. بعد ذلك ينشر SwitchState على /room_315/rails/right/switches/state.
  6. إذا publish_visual_switch_commands مفعلة، ينشر أيضاً string على /mfja/conveyor/switch_cmd حتى يتحرك model السويتش في Gazebo.

لماذا الحالة الفعلية مهمة؟

الشاتل لا يستخدم command الخام. عند نهاية segment يستخدم self.switch_states، أي الحالة الفعلية بعد delay. لهذا ممكن ترسل command الآن، لكن route لا يتغير إلا بعد تطبيق delay.

دور visual controller

  1. conveyor_loop_mode_controller.py يشترك على /mfja/conveyor/switch_cmd.
  2. عندما يستقبل A1R=EXTERIOR مثلاً، يغير yaw وcolor للـ model في Gazebo.
  3. ينشر ملخصاً على /mfja/conveyor/switch_states.
  4. rail nodes مشتركة اختيارياً بهذا الملخص. إذا بدأت node بعد أن Gazebo كان على وضع معين، تستطيع مزامنة route logic من الحالة البصرية.

6. منطق الـ stoppers

ros2 topic pub --once /room_315/rails/right/stoppers/command \
  mfja_rail_interfaces/msg/StopperCommand \
  "{stoppers: [{name: 'A1', state: '1'}]}"
  1. الرسالة تصل إلى _on_stopper_command().
  2. 1 أو STOP أو CLOSED تتحول إلى state stop. 0 أو PASS أو OPEN تتحول إلى state pass.
  3. _schedule_stopper_state_updates() يستخدم stopper_motion_delay_s.
  4. عندما تصبح الحالة actual، ينشر node StopperState.
  5. في loop الحركة، _active_stopper_ahead() يبحث عن stopper مغلق أمام الشاتل على نفس segment. إذا وجده، يوقف الشاتل قبل النقطة بمسافة stopper_stop_before_m.

حساس الاقتراب ليس مكاناً مستقلاً بعد التحديث الأخير. هو يتبع موقع stopper نفسه، ويبقى له فقط radius_m الخاص بالكشف.

7. منطق الشاتلات

إضافة شاتل

ros2 service call /room_315/rails/right/shuttles/add \
  mfja_rail_interfaces/srv/AddShuttle \
  "{name: 'room315_right_shuttle_1', start_slot: '2', speed: 0.2, start_enabled: true}"
  1. الطلب يصل إلى _on_add_shuttle_request().
  2. الكود يبحث عن start_slot داخل rail_devices_*.yaml.
  3. يحول slot إلى segment و s_ratio ثم ينشئ KinematicShuttleCore.
  4. إذا Gazebo spawn مفعّل، ينشئ model للشاتل أو يستخدم model موجود مسبقاً.

تشغيل وإيقاف شاتل

ros2 topic pub --once /room_315/rails/right/shuttles/command \
  mfja_rail_interfaces/msg/ShuttleCommand \
  "{name: 'ALL', command: 'ON'}"

_on_shuttle_control_command() يقبل: ON OFF RESET REMOVE

إذا الشاتل disabled، يبقى mode غالباً WAITING ولا يتقدم على السكة. إذا enabled، يدخل loop الحركة في كل tick.

8. اللوب الداخلي للحركة

_tick() هو heartbeat الرئيسي. يعمل حسب update_rate_hz. في كل tick يحدث التالي:

  1. يحسب dt من آخر tick.
  2. يحدث markers إذا كانت مفعلة.
  3. يطبق switch وstopper updates التي انتهى delay الخاص بها.
  4. لكل شاتل، يفحص إن كان deployed وenabled.
  5. يفحص stopper أمام الشاتل. إذا يوجد stopper مغلق أمامه، يوقفه عند مسافة أمان.
  6. يفحص collision avoidance مع الشاتلات الأخرى. إذا يوجد شاتل قريب، ينتظر.
  7. إذا لا يوجد مانع، يستدعي KinematicShuttleCore.step().
  8. step() يحرك الشاتل داخل segment. إذا وصل نهاية segment، يسأل routing_table عن successor حسب actual switch state.
  9. إذا لم يوجد successor صالح، يدخل الشاتل mode FALLING.
  10. ينشر pose للشاتل ويرسل pose إلى Gazebo إذا كانت service جاهزة.
  11. ينشر ShuttleState و SwitchState و StopperState.
tick apply pending states
guards stopper and collision checks
core.step move on current segment
routing choose next segment or FALLING
publish pose, states, Gazebo pose

9. منطق الحساسات

الحساسات ليست subscribers. هي feedback محسوب من موضع الشاتل الحالي. timer مستقل يعمل حسب sensor_publish_rate_hz وينشر SensorFeedback.

نوع الحساس في YAML كيف يصبح active؟ أين ينشر؟
position_sensors إذا كان أي شاتل على نفس segment وضمن radius_m من موقع الحساس. /room_315/rails/{side}/sensors/feedback
approach_sensors يتبع stopper مطابق. إذا صار شاتل ضمن radius من نقطة stopper، يصبح approach sensor active.

كل reading يحتوي: name, active, shuttle_name, segment, s, s_ratio.

10. التعامل مع Gazebo

المنطق الحقيقي للسكة داخل ROS node. Gazebo يستخدم للعرض والتفاعل البصري.

ما الذي يحدث كيف يصل إلى Gazebo؟
تحريك model الشاتل rail node ينشر PoseStamped ويستدعي service /world/<world>/set_pose إذا enable_gazebo_set_pose مفعلة.
إضافة أو حذف شاتل rail node يستخدم services /world/<world>/create و /world/<world>/remove.
إظهار markers للحساسات والـ stoppers rail node ينشئ models صغيرة في Gazebo ويحدث ألوانها حسب active/state.
تحريك السويتشات البصرية rail node ينشر /mfja/conveyor/switch_cmd، و conveyor_loop_mode_controller.py يطبق pose/color على models السويتشات.

11. منطق التحكم بالروبوتات

الروبوتات لا تتحكم مباشرة بمنطق السكة داخل room_315_kinematic_shuttle_node.py. هي طبقة مستقلة في multi_robot_sim.launch.py. الربط العملي بين الروبوت والسكة يكون عبر controller خارجي أو سكربت يقرأ حساسات السكة ثم يرسل أوامر للروبوت.

ما الذي يفعله launch الروبوتات؟

  1. يقرأ robots.yaml أو robots_room_315_only.yaml.
  2. يضيف model لكل روبوت إلى Gazebo في pose المحدد.
  3. يشغل robot_state_publisher لكل روبوت.
  4. ينشئ bridge بين ROS وGazebo لكل روبوت.
توبيك ROS الاتجاه الدور
/{robot_name}/joint_trajectory ROS إلى Gazebo ترسل trajectory للروبوت، مثل /kuka1/joint_trajectory.
/{robot_name}/joint_states Gazebo إلى ROS حالة joints الحالية من Gazebo.
/{robot_name}/joint_trajectory_progress Gazebo إلى ROS تقدم تنفيذ trajectory.
/{robot_name}/cmd_vel ROS إلى Gazebo فقط للروبوتات المتحركة مثل TIAGo base.
/{robot_name}/odom Gazebo إلى ROS فقط للروبوتات المتحركة.

المثال العملي: controller خارجي يراقب /room_315/rails/right/sensors/feedback. عندما يرى DZI1R.active=1، يرسل trajectory إلى /kuka1/joint_trajectory. هذا الربط غير موجود تلقائياً داخل rail node، بل تضيفه أنت في controller أعلى.

12. كيف تتتبع مشكلة؟

إذا السويتش لا يغير route

  1. انشر الأمر على /room_315/rails/right/switches/command.
  2. راقب /room_315/rails/right/switches/state.
  3. إذا state لم تتغير، افحص قيمة state: يجب أن تكون E/I أو EXTERIOR/INTERIOR.
  4. إذا state تغيرت لكن الشاتل يسقط، افحص routing_table للـ segment الحالي.

إذا السويتش يتحرك بصرياً لكن route لا يتغير

  1. راقب /mfja/conveyor/switch_states.
  2. راقب /room_315/rails/right/switches/state.
  3. إذا visual state فقط تغيرت، تأكد أن sync_from_visual_switch_states مفعلة أو أرسل command typed إلى rail node.

إذا حساس لا يصبح active

  1. راقب /room_315/rails/right/shuttles/state واعرف current_segment و s.
  2. افتح rail_devices_right.yaml أو rail_devices_left.yaml.
  3. تأكد أن الحساس على نفس segment وأن s_ratio قريب من موضع الشاتل.
  4. إذا المشكلة approach sensor، عدل stopper المقابل لأن approach sensor يتبع stopper.

إذا الروبوت لا يتحرك

  1. تأكد أن الروبوت مضاف في robots_room_315_only.yaml أو robots.yaml و enabled: true.
  2. راقب /{robot_name}/joint_states لترى إن كان Gazebo يعيد حالة joints.
  3. انشر trajectory على /{robot_name}/joint_trajectory.
  4. راقب /{robot_name}/joint_trajectory_progress.

13. ملخص سريع للحفظ

إذا رأيت هذا فكر بهذا
switches/command طلب تغيير. ليس بالضرورة actual بعد.
switches/state الحالة الفعلية التي يستخدمها route logic.
stoppers/state إذا stopper مغلق أمام الشاتل، الشاتل ينتظر قبل النقطة.
sensors/feedback feedback محسوب من موقع الشاتل، وليس message قادم من Gazebo.
shuttles/state.current_segment أهم قيمة لفهم أين الشاتل ولماذا حساس معين مفعل أو غير مفعل.
FALLING وصل الشاتل لنهاية segment ولم يجد successor صالح حسب routing والسويتش.
/mfja/conveyor/switch_cmd أمر مرئي للسويتشات في Gazebo، وليس هو topic التحكم الرئيسي بالسكة.
/{robot_name}/joint_trajectory مدخل التحكم بالروبوت عبر bridge إلى Gazebo.
